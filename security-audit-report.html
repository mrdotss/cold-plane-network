Findings

High H1 — No CSRF Origin Validation on Mutating Routes
app/api/auth/login/route.ts · app/api/auth/register/route.ts · app/api/auth/logout/route.ts
POST handlers don't validate the Origin or Referer header. SameSite=Lax protects against cross-site form submissions, but fetch() from a malicious page with certain browser configurations could bypass this.
Fix: Add Origin header check at the top of each POST handler: 
const origin = request.headers.get("origin");
if (origin && origin !== process.env.NEXT_PUBLIC_APP_URL) {
  return NextResponse.json({ error: "Forbidden" }, { status: 403 });
}
High H2 — No Rate Limiting on Registration Endpoint
app/api/auth/register/route.ts
checkRateLimit() is only called in the login route. An attacker can spam registrations to fill the database with junk accounts and exhaust storage.
Fix: Import and call checkRateLimit(ip) at the top of the register POST handler, identical to the login route pattern.
High H3 — No Username Input Sanitization
app/api/auth/register/route.ts
Username is passed directly from request.json() to Prisma without trimming, length validation, or character restrictions. Users can register with whitespace-only names, extremely long strings, or strings containing control characters.
Fix: Add validation before the Prisma call: 
const trimmed = username.trim();
if (trimmed.length < 3 || trimmed.length > 32
    || !/^[a-zA-Z0-9_-]+$/.test(trimmed)) {
  return NextResponse.json(
    { error: "Invalid username" }, { status: 400 }
  );
}
Medium M1 — Weak Password Policy
app/api/auth/register/route.ts · components/signup-form.tsx
Only checks length >= 8. The project's security.md spec requires at least one uppercase, one lowercase, and one digit. Neither server-side nor client-side enforces this.
Fix: Add regex validation: 
const strongPassword = /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).{8,}$/;
if (!strongPassword.test(password)) {
  return NextResponse.json({
    error: "Password needs uppercase, lowercase, and a digit"
  }, { status: 400 });
}
Medium M2 — In-Memory Rate Limiter Resets on Restart
lib/auth/rate-limit.ts
Uses a Map that's lost on server restart and doesn't work across multiple instances. Acceptable for MVP single-server, but an attacker can trigger a restart to reset all counters.
Fix: For production, migrate to Redis or a DB-backed rate limiter. For MVP, document the limitation.
Medium M3 — No Expired Session Cleanup
lib/auth/session.ts · prisma/schema.prisma
validateSession() deletes individual expired sessions on access, but there's no background job to prune stale sessions. The Session table will grow unbounded over time.
Fix: Add a periodic cleanup — either a cron job or a probabilistic cleanup on session validation (e.g., 1% chance to run deleteMany({ where: { expiresAt: { lt: new Date() } } })).
Medium M4 — Spoofable IP in Rate Limiter
app/api/auth/login/route.ts
Rate limiting keys on x-forwarded-for which is client-controllable. An attacker can rotate this header value to bypass the rate limit entirely.
Fix: Use the rightmost IP from a trusted proxy chain, or rely on a server-set header from your reverse proxy/load balancer.
Medium M5 — No Session Rotation on Login
app/api/auth/login/route.ts · lib/auth/session.ts
If a user logs in while already having a valid session, the old session remains active. No rotation on sensitive actions (password change). An attacker with a stolen token retains access even after the user re-authenticates.
Fix: On successful login, invalidate all existing sessions for the user before creating a new one: 
await prisma.session.deleteMany({ where: { userId: user.id } });
Low L1 — Fire-and-Forget Audit Logging
app/api/auth/login/route.ts
logFailure() uses .catch(() => {}). If the DB is down, failed login attempts are silently lost with no fallback.
Fix: Add a console.error fallback in the catch block so failures are at least visible in server logs.
Low L2 — No Content-Type Validation
app/api/auth/login/route.ts · app/api/auth/register/route.ts
Auth routes call request.json() without checking that the request has Content-Type: application/json. Malformed requests throw and return a generic 500 instead of a descriptive 400.
Fix: Check content type before parsing, or wrap request.json() in a try/catch that returns 400.
Low L3 — XSS in Error Display (Safe)
components/login-form.tsx · components/signup-form.tsx
Error state is rendered directly in JSX via {error}. React auto-escapes string interpolation, so this is safe. No use of dangerouslySetInnerHTML found. Noting for completeness — no action needed.
